# Переписывание rakurs-site на Astro — дизайн

**Дата:** 2026-07-13
**Статус:** утверждено (обсуждение в сессии Claude Code)

## Цель

Переписать сайт Rakurs с монолитного `index.html` (~1950 строк) + самописного
рантайма (`support.js`, dc-runtime) на Astro, чтобы код стал читаемым,
поддерживаемым и масштабируемым, а SEO настраивалось из одного места.
Дизайн и контент переносятся 1:1 — это техническое переписывание, не редизайн.

## Контекст: проблемы текущего кода

1. **Монолит** — весь сайт в одном `index.html` (197KB): инлайн-стили в
   атрибутах, несколько `<style>`-блоков, JSON-LD, скрипты, шаблон компонента.
2. **Самописный фреймворк без исходников в репо** — `support.js` (1687 строк)
   скомпилирован из `dc-runtime/src/*.ts`, которых в репозитории нет
   («GENERATED — do not edit»). React берётся из `window` (CDN).
   Интерактив фактически неправимый.
3. **Хрупкий i18n** — EN-версия делается в рантайме подменой текстов по
   словарю «русская строка → английская» (`i18n.js`, 55KB). Правка русского
   текста молча ломает перевод; EN-контент существует только после выполнения
   JS; hreflang указывает на `?lang=en` (индексируется хуже отдельных URL).
4. **Контент зашит в код в четырёх местах** — продукты B1–B6, кейсы и FAQ
   продублированы: JS-массивы в инлайн-скрипте, JSON-LD, `llms.txt`,
   `llms-full.txt`. Синхронизация ручная.
5. **Домен захардкожен** — URL GitHub Pages в ~10 местах, миграция через sed.

Сама SEO/GEO-работа (мета, JSON-LD, FAQ, llms.txt, robots) сделана хорошо и
переносится полностью.

## Решения (зафиксированы с пользователем)

- **Стек:** Astro (статический вывод). Рассматривались Next.js static export
  (тяжелее, гидратация везде, i18n в статике через костыли) и 11ty/Vite
  (всё SEO руками, свой мини-фреймворк — та же проблема, что сейчас).
- **Хостинг:** Gcore — Object Storage (S3-совместимый) + CDN. Чистая статика,
  без серверной части.
- **Рост:** отдельные страницы продуктов (`/products/b1` …) и кейсов
  (`/cases/insurance` …).
- **Контент правит** владелец сам + AI-инструменты → Markdown/JSON в репо,
  CMS не нужна.

## Архитектура

### Структура проекта

```
src/
  content/                  # весь контент отдельно от кода
    products/  b1.ru.md, b1.en.md, … b6.*.md
    cases/     insurance.ru.md, insurance.en.md, …
    faq/       *.ru.md, *.en.md
  components/               # секции лендинга и переиспользуемые блоки
    Hero.astro, Nav.astro, Pain.astro, Storefront.astro,
    ProductCard.astro, Cases.astro, Founders.astro, Faq.astro,
    LeadForm.astro, Footer.astro, …
  layouts/
    Base.astro              # единственное место SEO
  pages/
    index.astro
    products/[slug].astro   # → /products/b1 …
    cases/[slug].astro      # → /cases/insurance …
    en/…                    # зеркальные EN-страницы
    llms.txt.ts             # генерация из контента
    llms-full.txt.ts
  styles/
    tokens.css              # цвета, шрифты, отступы (CSS custom properties)
    global.css
  scripts/                  # интерактив, обычный TypeScript
    chat.ts, pipe.ts, calculator.ts, form.ts, …
public/
  robots.txt, og.png, rakurs-team.jpg, favicon.svg
```

### Единый источник правды для контента

Content-коллекции Astro с zod-схемами. Схема продукта: code, tag, hole,
title, desc, gets[], price, launch, cta, note, demo. Схема кейса: tab, num,
before, after, metric, metricLabel, note. Невалидный контент ломает билд.

Из коллекций на билде генерируются: HTML-страницы, JSON-LD
(ProfessionalService-каталог, FAQPage), `llms.txt`, `llms-full.txt`, sitemap.
Правка цены в одном файле обновляет всё.

### SEO-слой

Один компонент в `Base.astro` принимает title/description/локаль/слаг и
выводит: `<title>`, description, canonical, пары hreflang ru↔en↔x-default,
OG/Twitter-мета, JSON-LD (Organization, WebSite — глобально; остальное по
страницам). Домен — одна константа `site: 'https://rakurs.ai'` в
`astro.config.mjs`; до переезда — текущий GitHub Pages URL. Sitemap —
`@astrojs/sitemap` с i18n-связками. robots.txt — статика в `public/`,
разрешения для LLM-ботов сохраняются. Слот для аналитики (GA4/GoatCounter)
в `Base.astro` через переменную окружения.

### i18n

Встроенный i18n-роутинг Astro: RU — корень (`defaultLocale`, без префикса),
EN — `/en/…`. Переводы — параллельные файлы контента и словарь UI-строк
(`src/i18n/ru.ts`, `en.ts`) для мелких подписей. Рантайм-подменятель и
`i18n.js` умирают. Переключатель языка — ссылка на парную страницу.
Существующие переводы из `i18n.js` используются как исходник при разносе
контента по EN-файлам. `?lang=` больше не используется; EN-версия — честная
статика.

### Интерактив

Виджеты (чат-демо, «труба» с деньгами, калькулятор потерь, форма, scrollspy,
reveal-анимации) переписываются с dc-runtime на обычные TypeScript-модули,
подключаемые `<script>` внутри своих компонентов. Без React, без CDN-зависимостей.
Исходники живут в этом репозитории. Поведение переносится 1:1 из `support.js`
(он остаётся читаемым референсом до конца миграции).

### Стили

Дизайн переносится 1:1. Цвета (#E37722, #1B1815, #F9F5EC, #F3EEE3, #E4DBCD …),
шрифты, отступы — в `tokens.css` как CSS-переменные. Стили секций — scoped
CSS в `.astro`-компонентах вместо инлайн-атрибутов. Без Tailwind.

### Формы

FormSubmit как сейчас (POST со статической страницы). Логика
валидации/состояний — в `scripts/form.ts`.

### Деплой

GitHub Actions: `npm run build` → `dist/` → выгрузка в Gcore Object Storage
(S3-совместимый API), поверх — Gcore CDN. До готовности Gcore допустим
текущий GitHub Pages (dist отдаётся как статика).

## Обработка ошибок

- Невалидный контент (пропущенное поле, кривой frontmatter) → билд падает
  с указанием файла (zod).
- `astro check` — типы TS в компонентах и скриптах.
- Битые внутренние ссылки RU↔EN проверяются на билде (страницы генерируются
  из одних коллекций — рассинхрон слагов виден сразу).

## Тестирование

- CI: `astro check` + `astro build` на каждый пуш.
- После порта: Rich Results Test (JSON-LD), Lighthouse (CWV должны вырасти —
  ноль JS по умолчанию против текущих ~120KB скриптов), ручная сверка
  секций со старым сайтом бок о бок, проверка EN-страниц без JS
  (curl / отключённый JS).

## Миграция по шагам

Каждый шаг оставляет сайт собираемым и рабочим:

1. Каркас Astro + tokens.css + Base.astro с SEO-компонентом.
2. Перенос секций лендинга в компоненты (RU, статика, без интерактива).
3. Контент → коллекции; JSON-LD, llms.txt, sitemap — генерация.
4. EN-страницы через i18n-роутинг (переводы из `i18n.js` → файлы контента).
5. Интерактивные виджеты: dc-runtime → `scripts/*.ts`.
6. Новые страницы `/products/[slug]`, `/cases/[slug]` (RU+EN).
7. CI/деплой на Gcore, затем переезд домена на rakurs.ai
   (замена одной константы + DNS + Search Console, по DOMAIN_MIGRATION.md).

Старые файлы (`index.html`, `support.js`, `i18n.js`) удаляются в конце,
после сверки. История остаётся в git.

## Вне рамок (YAGNI)

- Редизайн, изменение текстов и структуры секций.
- CMS, блог, дополнительные локали (архитектура их допускает, но не строим).
- Серверный бэкенд для чата (демо-сценарий остаётся скриптовым).
- Per-page og-изображения (satori) — возможное будущее улучшение.
